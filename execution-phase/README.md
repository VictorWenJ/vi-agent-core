# Execution Phase 阶段文档治理说明

## 1. 文档定位

`execution-phase/` 是本项目用于管理各开发阶段详细设计、开发计划、测试验收、Codex 执行提示词和阶段收口记录的目录。

本目录的核心目标是：

- 将阶段性详细设计从根目录治理文档中下沉出来；
- 避免根目录文档在每个阶段反复更新后发生内容漂移；
- 保留每个阶段的设计痕迹、执行痕迹和验收痕迹；
- 让每个阶段都有独立、清晰、可追溯的事实源；
- 降低 ChatGPT / Codex 在长周期项目中因上下文限制导致的遗漏和误改风险。

从本目录建立后，项目文档治理原则调整为：

> 根目录文档负责长期原则，阶段目录文档负责阶段细节。

---

## 2. 文档职责边界

本项目采用“文档治理驱动开发”的方式。

但是，阶段性详细设计不应持续堆叠到根目录文档中。

文档职责划分如下：

| 文档区域 | 职责 |
|---|---|
| 根目录 `AGENTS.md` | 项目级协作规则、通用开发约束、文档治理规则、强契约原则 |
| 根目录 `ARCHITECTURE.md` | 长期架构分层、模块边界、系统总体方向 |
| 根目录 `CODE_REVIEW.md` | 通用 code review 原则、通用反回退检查项 |
| 根目录 `PROJECT_PLAN.md` | 高层路线图、当前阶段索引、阶段状态 |
| 模块级 `AGENTS.md` | 模块职责、包边界、本模块长期开发约束 |
| `execution-phase/{phase}/` | 某一具体阶段的详细设计、开发计划、测试验收、Codex prompt、阶段收口 |

根目录文档应保持稳定、低频更新。  
阶段目录文档允许随阶段推进进行详细设计和执行记录。

---

## 3. 阶段目录命名规范

每个阶段目录应使用明确的阶段编号和阶段主题命名。

推荐格式：

```text
phase-{阶段编号}-{阶段主题}
```

示例：

```text
phase-P1-logical-distributed-baseline
phase-P1.5-project-governance
phase-P2-context-memory
phase-P2-E-prompt-governance
phase-P2.5-architecture-governance
phase-P3-graph-workflow-kernel
```

阶段目录名应满足：

- 能清楚表达阶段编号；
- 能清楚表达阶段主题；
- 不使用过于抽象的名称，例如 `phase/current/temp/new`；
- 阶段一旦完成，不应随意重命名；
- 如确需重命名，必须同步更新引用路径和阶段索引。

---

## 4. 标准阶段目录结构

从本目录建立后，每个新阶段建议使用以下标准结构：

```text
execution-phase/{phase-name}/
├── README.md
├── design.md
├── plan.md
├── test.md
├── prompts.md
└── closure.md
```

各文件职责如下：

| 文件 | 职责 |
|---|---|
| `README.md` | 当前阶段入口说明、阶段目标摘要、阅读顺序、关键文件索引 |
| `design.md` | 当前阶段详细系统设计、领域对象、包边界、强契约、流程设计、不做事项 |
| `plan.md` | 当前阶段开发计划、任务拆分、输入输出、修改范围、执行顺序、完成标准 |
| `test.md` | 当前阶段 code review 清单、回归测试范围、反回退检查项、验收标准 |
| `prompts.md` | 当前阶段给 Codex 的正式 prompt、执行目标、执行结果摘要 |
| `closure.md` | 当前阶段最终收口记录、完成范围、遗留事项、是否需要晋升到根文档 |

历史阶段目录如果尚未完全符合该结构，可以暂时保留原状。  
后续可在项目治理阶段中逐步补齐，不应为了整理历史文档而打断当前阶段开发。

---

## 5. 当前阶段索引规则

当前活跃阶段不在本文件中写死。

当前阶段应以根目录 `PROJECT_PLAN.md` 中的阶段状态为准。

`PROJECT_PLAN.md` 应以索引方式指向当前阶段目录，例如：

```text
P2-E：进行中，详见 execution-phase/phase-P2-E-prompt-governance/
```

本文件只定义阶段文档体系的长期规则，不承载某个具体阶段的详细设计。

这样可以避免本文件在每个阶段切换时被反复大改。

---

## 6. 历史强契约规则

历史阶段文档如果已经被确认为强契约，不应因为新阶段开始而失效。

对于已经冻结的历史强契约：

- 仍然对其覆盖范围内的已有对象、已有语义、已有流程具备约束力；
- 新阶段不得随意覆盖历史强契约中的已有对象定义；
- 如果新阶段需要修改历史强契约覆盖的对象，必须明确说明这是契约升级，而不是普通实现变更；
- 契约升级必须先修改对应阶段设计文档或新增阶段设计文档，再修改代码。

如果历史强契约和当前阶段设计发生冲突：

- 已有对象优先遵守历史强契约；
- 新增对象优先遵守当前阶段 `design.md`；
- 无法判断时，必须先修正文档，再继续实现；
- 不允许 Codex 或开发者用工程习惯自行推断、改名、扩展字段。

---

## 7. 文档优先级规则

当不同文档之间存在重叠或冲突时，按照以下优先级判断：

```text
1. 当前阶段 execution-phase/{phase}/design.md 中明确声明的新增阶段强契约
2. 当前阶段 execution-phase/{phase}/plan.md 和 test.md
3. 已冻结的历史阶段强契约文档
4. 根目录 AGENTS.md / ARCHITECTURE.md / CODE_REVIEW.md / PROJECT_PLAN.md
5. 模块级 AGENTS.md
```

补充规则：

- 根目录文档是长期治理源；
- 阶段目录文档是阶段事实源；
- 模块级文档是模块边界源；
- `prompts.md` 是执行痕迹源；
- `closure.md` 是阶段收口源。

---

## 8. 阶段文档冻结规则

每个阶段完成后，应通过 `closure.md` 进行收口。

阶段完成后：

- `design.md` 不应被随意重写；
- `plan.md` 不应被反复改造成新阶段计划；
- `test.md` 不应被删除关键验收项；
- `prompts.md` 应保留真实执行过的 Codex prompt；
- `closure.md` 应记录最终完成范围、遗留事项和是否晋升根文档。

如果阶段完成后发现设计需要补充，不建议直接重写原始设计内容。  
推荐在阶段文档末尾追加：

```text
Design Amendments
```

或：

```text
Post-closure Notes
```

用于保留文档演进痕迹。

---

## 9. 根目录文档更新规则

阶段开发过程中，不应频繁大改根目录文档。

根目录文档只在以下情况更新：

1. 某个阶段产出的规则已经成为长期通用规则；
2. 某个阶段引入了稳定的新架构层；
3. 某个阶段改变了长期模块边界；
4. 当前阶段状态需要在 `PROJECT_PLAN.md` 中更新；
5. 文档优先级、协作规则、review 总原则发生变化。

阶段细节不应直接写入根目录文档。

例如：

- 阶段详细 class 字段不写入根目录 `ARCHITECTURE.md`；
- 阶段完整执行拆分不写入根目录 `PROJECT_PLAN.md`；
- 阶段专项 review 清单不写入根目录 `CODE_REVIEW.md`；
- 阶段 Codex prompt 不写入根目录 `AGENTS.md`。

这些内容应写入对应的：

```text
execution-phase/{phase-name}/
```

---

## 10. Codex 执行前阅读顺序

从本目录建立后，Codex 执行阶段任务前必须按顺序阅读：

```text
1. 根目录 AGENTS.md
2. 根目录 PROJECT_PLAN.md
3. 根目录 ARCHITECTURE.md
4. 根目录 CODE_REVIEW.md
5. execution-phase/README.md
6. 当前阶段 README.md
7. 当前阶段 design.md
8. 当前阶段 plan.md
9. 当前阶段 test.md
10. 与当前任务相关的历史强契约文档
11. 相关模块 AGENTS.md
12. 相关源码与测试
```

如果任务涉及已有历史阶段对象，必须同步阅读对应历史阶段的强契约文档。

如果任务涉及新增阶段对象，必须以当前阶段 `design.md` 为新增契约源。

---

## 11. 阶段开发标准流程

每个阶段建议按以下流程推进：

```text
1. 新建阶段目录
2. 编写阶段 README.md
3. 编写 design.md
4. 编写 plan.md
5. 编写 test.md
6. 在 prompts.md 中记录 Codex prompt
7. Codex 执行实现
8. 上传完整源码快照
9. 按 test.md 进行静态验收
10. 如有问题，追加新的 Codex prompt 到 prompts.md
11. 阶段完成后填写 closure.md
12. 判断是否需要将长期规则晋升到根目录文档
13. 冻结当前阶段文档
```

---

## 12. `prompts.md` 使用规则

每次给 Codex 的正式 prompt 都应记录到当前阶段的 `prompts.md` 中。

记录内容建议包括：

- prompt 编号；
- 对应小阶段；
- 执行目标；
- 完整 prompt；
- 执行结果摘要；
- 是否需要二次修复。

`prompts.md` 的作用是保留执行痕迹，不是替代 `design.md` 或 `plan.md`。

---

## 13. `closure.md` 使用规则

阶段完成后必须填写 `closure.md`。

`closure.md` 应至少记录：

- 本阶段最终完成内容；
- 实际修改文件；
- 最终架构决策；
- 延后事项；
- 回归测试结果；
- 是否需要晋升根目录文档；
- 最终验收结论。

没有 `closure.md` 收口记录，不应宣布阶段正式完成。

---

## 14. 禁止事项

`execution-phase/` 建立后，禁止以下行为：

1. 不允许继续把阶段详细设计长期堆叠到根目录 `ARCHITECTURE.md`；
2. 不允许把阶段专项 review 清单长期堆叠到根目录 `CODE_REVIEW.md`；
3. 不允许把阶段完整开发计划长期堆叠到根目录 `PROJECT_PLAN.md`；
4. 不允许让 Codex 在没有阅读当前阶段文档的情况下直接实现；
5. 不允许为了旧测试保留过时架构逻辑；
6. 不允许阶段完成后删除关键设计痕迹；
7. 不允许新旧阶段语义混写在同一个历史设计文档中；
8. 不允许在没有 closure 记录的情况下宣布阶段完成；
9. 不允许用工程习惯替代冻结文档中的 class 契约；
10. 不允许在未明确文档优先级的情况下处理冲突。

---

## 15. 总结

`execution-phase/` 是项目阶段事实源。

根目录文档负责长期治理。  
模块级文档负责模块边界。  
阶段目录文档负责阶段细节。  
`prompts.md` 负责执行痕迹。  
`closure.md` 负责阶段收口。

本文件只定义阶段文档体系的长期规则，不承载某个具体阶段的详细设计。
