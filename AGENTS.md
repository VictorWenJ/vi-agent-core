## 2. 项目定位

`vi-agent-core` 是一个基于 Java 的 Agent Runtime Framework 实现，不是普通聊天后端。

项目目标：
- 构建一套可交互、可执行、可扩展、可评估、可审计的 Agent 运行时底座；
- 以 Java 工程方式沉淀 Runtime、Tool、Provider、Transcript、Observability、State Memory 等稳定边界；
- 为后续企业级 AI 工作台、留学/移民顾问工作台等产品形态提供核心后端。

### 2.1 模块结构

仓库包含 5 个 Maven 模块：

- `vi-agent-core-app`：顶层 Spring Boot 启动、WebFlux 接入、配置装配、SSE 适配，**唯一可运行模块**
- `vi-agent-core-runtime`：Runtime 主链路编排、Loop、Context、Tool 协调、运行时事件、运行时记忆接口与指标抽象
- `vi-agent-core-infra`：Provider、Persistence、Mock Integration 等基础设施实现
- `vi-agent-core-model`：内部运行时模型与跨层共享契约承载层
- `vi-agent-core-common`：异常、ID、通用无状态工具等轻量公共能力

**标准目标依赖方向**：

- `common`：最底层，保持轻量
- `model` → `common`
- `runtime` → `model` + `common`
- `infra` → `model` + `common`
- `app` → `runtime` + `infra` + `model` + `common`

## 3. 仓库级通用开发规范（长期有效）

> 本节全部属于**通用、长期生效**的开发规范。后续无论哪个模块开发，只要没有被新的根文档正式替换，都必须继续遵守。  
> 本节在不脱离本项目分层与模块边界的前提下，吸收 Oracle / OpenJDK 官方公开规范中的通用要求，包括：包命名与层次组织、命名约定、编程实践、Javadoc / doc comments、`Optional` 与 `Objects` 的语义化使用、以及 `var`（局部变量类型推断）的官方风格建议。

### 3.1 总体分层规范
- `common`：只放共享异常、ID、基础工具、纯通用常量；不得放业务对象和业务逻辑。
- `model`：只放领域模型、值对象、枚举、端口/接口；不得依赖 `runtime` / `infra` / `app`。
- `runtime`：只放运行时编排、业务规则、上下文工程、tool orchestration（工具编排）；只允许依赖 `model` + `common`。
- `infra`：只放外部系统适配实现、provider、DB/Redis repository、外部资源 repository 实现；只允许依赖 `model` + `common`。
- `app`：只放 controller、application service、Spring 配置与装配；允许依赖所有下层模块。
- 禁止反向依赖；尤其禁止 `runtime -> infra` 与 `infra -> runtime`。
- 分层的目标不是“代码能编译”，而是职责清晰、依赖方向稳定、替换边界明确。

### 3.2 包与目录规范
- 包名必须全小写，采用反向域名风格。
- 包路径必须表达真实职责，禁止使用 `util`、`helper`、`misc`、`temp`、`test2` 等语义模糊或临时性目录名长期承载正式代码；本项目允许的公共工具目录以 `src/main/java/com/vi/agent/core/common/util` 为权威公共工具目录。
- 目录划分必须优先服务于业务边界和依赖边界，而不是仅按个人习惯组织代码。
- 中大型项目优先按业务模块划分，再在模块内部按技术职责分层；禁止全局大平铺式堆放所有 `controller`、`service`、`repository`、`dto`。
- 同一目录下对象必须职责同类；`controller`、`domain`、`repository`、`config`、`dto`、`entity` 不得长期混放。
- `port`、`adapter`、`client`、`repository` 等目录名必须与实际职责严格一致；禁止目录语义与类职责不符。
- `port` 包只放接口；值对象、结果对象、DTO、command、record 不得放入 `port`。
- DTO、Entity、VO、Command、Query、Event 等不同语义对象应按职责分目录管理，不得长期混放。
- 包位置必须表达真实职责；如果类名和包位置语义冲突，优先调整包位置，而不是继续容忍语义错位。
- 生产代码中的 `entity` / `dto` / `command` / `event` / `domain object` 默认独立成类，不以内部类形式长期存在。
- 包结构必须服务于单向依赖关系；禁止通过目录组织掩盖反向依赖、跨层调用或职责越界。
- 公共包只允许放稳定、通用、跨模块复用的内容；禁止将无明确归属的类下沉到 `common` 形成垃圾场。
- 目录层级应保持适中，既要避免职责混杂，也要避免过度嵌套影响理解、检索与维护。
- 包级说明、包级注解、包级文档应优先使用 `package-info.java` 承载；不得把正式包说明散落到无约束注释中。

### 3.3 Maven / POM 规范
- 根 `pom.xml` 必须是聚合父 POM，统一声明 `<modules>`、`dependencyManagement`、`pluginManagement` 与仓库级构建规则。
- 所有第三方依赖版本必须统一由根 POM 的 `dependencyManagement` 管理；子模块禁止各自声明版本造成漂移。
- 所有公共构建插件版本必须统一由根 POM 的 `pluginManagement` 管理；子模块只声明实际需要使用的插件，不重复写版本。
- 子模块禁止自行写依赖版本，除非是临时验证依赖；验证结束后必须回写父 POM 并移除子模块中的临时声明。
- 子模块禁止显式写默认的 `<scope>compile</scope>`。
- 只有应用模块允许使用 `spring-boot-maven-plugin` 或其他可执行打包插件；库模块禁止生成可执行包。
- 所有模块必须遵守 Maven 标准目录布局：`src/main/java`、`src/main/resources`、`src/test/java`。
- 模块 POM 只保留直接依赖，不得通过“顺手加一个依赖”破坏分层边界或制造隐式耦合。
- 依赖引入必须以模块职责和分层关系为前提；禁止为了省事跨层引入上层模块依赖。
- 传递依赖不可作为正式设计前提；模块需要什么依赖，必须显式声明，不得长期“蹭传递依赖”。
- 禁止在多个子模块重复复制相同依赖声明而不做统一治理；公共依赖策略必须回收到父 POM 管理。
- `optional`、`exclusion`、`scope` 等配置必须有明确语义依据；禁止为了“先编过”而随意添加。
- 测试依赖只允许出现在测试作用域，不得将 `test` 工具链带入生产编译链路。
- 仓库、镜像源、插件仓库等配置应优先通过企业统一环境或父级治理控制；项目 POM 不得随意硬编码与仓库职责无关的外部源。
- SNAPSHOT、临时本地包、未稳定版本依赖不得长期存在于主干；进入正式提交前必须收口为可追踪、可复现的稳定版本。
- 模块命名、`artifactId`、`parent` 继承关系必须清晰稳定，能够直接反映模块职责，禁止出现 `xxx-new`、`xxx-temp`、`xxx-final2` 这类临时命名长期存在。
- POM 不只是让项目能编译；它还必须准确表达模块职责、依赖方向、构建边界与治理规则。

### 3.4 命名规范
- 类名必须使用大驼峰命名，且应为名词或名词短语，语义完整清晰，不使用含糊缩写、拼音缩写或无业务含义的占位命名。
- 接口名必须体现能力边界或抽象职责，不得仅以 `I` 前缀、`Base`、`Common`、`Service` 等空泛形式命名。
- 实现类命名必须准确体现其实现语义；只有在确有多实现语境时，才使用 `Default`、`Local`、`Remote`、`File`、`Db`、`Redis`、`OpenAi` 等前后缀区分实现。
- 方法名必须使用动词或动宾短语，准确表达动作；查询、校验、转换、构建、持久化、发布、装配等语义必须在方法名中体现清楚。
- 布尔字段和布尔方法必须直接表达真假语义，优先使用 `enabled`、`deleted`、`valid`、`success`、`isXxx`、`hasXxx`、`canXxx` 等形式，禁止使用 `flag`、`type`、`status` 代替布尔语义。
- 常量名必须全部大写并使用下划线分隔，名称应直接表达业务含义，禁止使用 `NUM1`、`DEFAULT_VALUE`、`TEMP_CONST` 这类弱语义命名。
- 包名、目录名、配置名、Bean 名、工具名、错误码名、异常名必须避免双重语义、历史残留命名和“看起来能用但实际含义不清”的命名方式。
- DTO、Command、Query、Request、Response、Entity、VO、Event 等对象后缀必须与对象真实职责一致；禁止混用或为了省事全部命名成 `DTO`。
- Repository、Client、Adapter、Manager、Handler、Resolver、Assembler、Factory、Provider、Executor 等后缀只有在职责真实匹配时才允许使用，禁止为了“显得专业”滥加后缀。
- 枚举名必须是业务概念名，枚举项必须使用全大写加下划线，且名称应直接表达业务语义；禁止使用 `TYPE1`、`STATUS_A`、`VALUE_B` 等无语义命名。
- 错误码命名必须单义、稳定、可扩展，能够体现对象、动作和失败语义；禁止使用 `SYSTEM_ERROR_1`、`FAIL_ERROR`、`UNKNOWN_BIZ_ERROR` 这类低质量命名。
- 异常类命名必须体现业务对象与异常原因，通常使用 `XxxNotFoundException`、`XxxValidationException`、`XxxConflictException`、`XxxExecutionException` 等清晰形式。
- 测试类命名必须直接对应被测对象与测试意图，单元测试、集成测试、契约测试、回归测试应通过名称清晰区分。
- 禁止出现 `New`、`Old`、`Temp`、`Tmp`、`Bak`、`Final2`、`TestHandle`、`ManagerImpl2` 等临时性或历史残留命名长期存在。
- 命名的首要目标是让阅读者在不跳转上下文的情况下理解对象职责；如命名与实际职责冲突，优先重命名，不以注释长期补救。

### 3.5 类设计规范
- 一个类只负责一个主职责；如果一个类承担 4 类以上明显不同职责，默认必须评估拆分。
- 类设计必须优先保证职责内聚、边界清晰、依赖可控，不得为了“先跑通”长期堆叠多种职责。
- 超过 300 行或持续膨胀的核心类，必须主动评估职责拆分，不得以“当前还能维护”为理由长期放任。
- 编排类不做 mapper 细节；repository 不做业务决策；provider 不做跨层流程编排；注册表不承担与注册无关的宽职责。
- 类内方法必须按功能分组，避免构造、查询、写入、转换、日志、异常处理、工具方法交叉混写。
- 类的公开接口必须尽量收敛，只暴露真实需要的能力；不得将内部实现细节以 `public` 形式泄漏给外部调用方。
- 一个类中的字段必须服务于同一主职责；若字段明显分属多个子领域，通常说明该类已发生职责漂移。
- 预留协作者与正式协作者必须明确区分，避免“先塞进类里以后再说”的宽松设计长期存在。
- 优先组合而非继承；只有在明确满足稳定的 `is-a` 关系时，才允许使用继承建模。
- 抽象类只用于沉淀稳定共性；若只是为了复用几段代码，不得滥用继承层级。
- 不允许通过“万能基类”“超大上下文父类”“统一结果父对象”等方式掩盖真实建模问题。
- 领域对象、配置对象、持久化对象、协议对象、运行时对象必须按职责分开建模，不得在单个类中混合承担多种层次语义。
- 类之间的协作关系必须清晰稳定；一个类若直接依赖过多外部协作者，必须评估是否应引入中间抽象或拆分职责。
- 状态持有类与纯工具类必须严格区分；工具类不得偷偷持有可变状态，状态类也不得伪装成工具类。
- 工具类必须是无状态、纯辅助性质；若开始承载业务流程、状态流转或跨对象协调，应升级为正式服务类。
- 类的构造完成后应尽量保持对象处于合法、自洽状态；禁止允许“半初始化对象”长期在主链路中流转。
- 值对象应优先不可变，并通过构造约束保证自身合法性；实体对象的可变边界也必须显式收紧。
- 如类名、包位置、字段集合、方法集合之间出现明显语义冲突，优先视为设计问题并整改，不长期容忍“名不副实”的类存在。
- 若某段逻辑已经在两个以上正式类中重复出现，默认必须评估是否抽取为公共方法、公共组件或公共协作者；相同逻辑出现第三次时不得继续复制粘贴。

### 3.6 方法设计规范
- 一个方法只负责一个清晰职责，避免同时承担校验、转换、查询、状态变更、持久化等多种行为。
- 方法职责边界必须清晰；方法名、参数、返回值和副作用必须一致，不得“看起来做 A，实际还顺手做 B / C / D”。
- 方法名必须准确表达动作语义，通常使用动词或动宾短语命名。
- 公共方法的参数和返回值必须具备明确语义，避免使用 `Object`、`Map<String, Object>`、裸 `List` 等弱类型长期承载正式业务含义。
- 方法参数数量过多时，应优先通过参数对象、构建器或拆分职责降低复杂度。
- 方法应优先保证输入输出清晰、调用方式直观，避免通过多个布尔参数、位置参数或 `null` 组合表达复杂语义。
- 公共方法必须进行必要的参数校验，避免将非法输入留给下游逻辑放大。
- 查询方法与修改方法应严格区分，避免“看起来是查询，实际有副作用”或“看起来是执行，内部又承担查询拼装”。
- 返回值设计必须稳定、可理解，不得混用 `null`、魔法值、整数状态码、字符串标记表达不同结果语义。
- 方法副作用必须可预期；调用方应能从方法命名和签名判断该方法是否会修改状态、写入数据或触发外部调用。
- 方法内部流程应保持层次清晰；嵌套过深、分支过多、循环体过重时，必须拆分为更小的方法。
- 条件判断应围绕明确规则组织，避免堆砌大量难以理解的 `if-else` 或重复分支。
- 转换类方法只做转换，不混入业务决策、持久化或外部调用。
- 工具类方法应保持无状态、纯辅助特征，避免偷偷持有上下文或产生隐蔽副作用。
- 私有方法也必须保持清晰语义与合理粒度，不得把私有方法当作杂项逻辑堆放区。
- 异常处理边界必须明确；方法要么在本层处理并转换语义，要么向上抛出，不得吞异常或只记录日志后继续执行。
- 事务边界应放在合适的方法层级，避免事务范围过小导致语义破碎，或范围过大导致耦合失控。
- 重复逻辑应抽取复用；同类重复逻辑出现 2 处及以上时，必须评估并优先收口为公共方法、公共组件或模板化步骤，禁止无节制复制粘贴。
- 方法设计应优先追求可读性、可测试性和可维护性，而不是单纯压缩代码行数或炫技式写法。

### 3.7 枚举与常量规范
- 对于有限、稳定、可穷举的业务集合，条件允许且语义匹配时，必须优先使用枚举表达，而不是散落的字符串常量、整数常量或布尔开关。
- 枚举只用于表达稳定业务语义；若取值集合频繁变更、依赖数据库配置或需要运行时扩展，不得硬编码为枚举。
- 涉及数据库落库、接口协议、跨服务传输时，不得直接依赖 `name()` 或 `ordinal()` 作为外部值；必须定义稳定字段，如 `code`、`value`。
- 严禁使用 `ordinal()` 参与业务判断、持久化或协议传输。
- 对外存在编码值的枚举，必须提供统一反查方法，如 `fromCode`、`ofCode`；不得在各业务处手写遍历查找。
- 枚举字段必须优先保持不可变，通常使用 `final` 修饰，并通过构造方法一次性初始化。
- 枚举不得承载复杂业务流程、数据库访问、远程调用或框架注入逻辑。
- 常量只用于稳定、简单、无状态的固定值；若某组常量已经具备明显业务枚举语义，应升级为枚举，而不是继续扩展常量类。

### 3.8 可变性与对象设计规范
- 领域对象优先不可变；构造完成后尽量不再通过裸 setter 改写核心状态。
- 对外暴露的集合必须返回只读视图，禁止外部直接拿到可变内部集合。
- 禁止在核心上下文对象上使用类级别 `@Setter`。
- 需要状态变化时，优先提供显式意图方法，而不是暴露一组无约束裸 setter。
- 核心对象的必要字段应在构建时补齐，不允许把“对象先创建，后面再慢慢补字段”当成默认模式。
- `record` 仅用于语义明确、数据承载型、浅不可变的小对象；若对象需要复杂行为、继承约束或框架特殊处理，必须评估是否仍适合使用 `record`。

### 3.9 Lombok 使用规范
- 简单 entity / dto / config properties 可以使用 Lombok。
- 核心领域对象禁止因为图省事而无脑上 `@Data`。
- 优先使用 `@Getter` + `@Builder`，谨慎使用 `@Setter`。
- 只要 Lombok 注解能完整等价替代手写样板代码（getter / setter / constructor / builder），必须优先使用注解，不得新增手写重复实现。
- 对需要业务约束的对象，不要只靠 Lombok 生成一堆无语义入口。
- 日志优先使用 `@Slf4j`；不要手动重复声明 logger 模板代码。
- 使用 Lombok 后，代码可读性不得下降；若对象创建路径、默认值来源、构造约束因此变得模糊，应回退为显式写法。

### 3.10 工具类与 JDK / Commons API 使用规范
- 在不破坏边界、不降低可读性的前提下，只要 `src/main/java/com/vi/agent/core/common/util` 目录下已有等价公共工具能力，必须优先复用，不得重复造轮子。
- 字符串判空、空白判断、截断、大小写比较、前后缀判断、默认值处理等，条件允许且语义匹配时，必须优先使用 `StringUtils` 相关 API，而不是重复手写 `null` + `trim` + `length` 组合判断。
- 集合判空、集合为空判断等，条件允许且语义匹配时，必须优先使用 `CollectionUtils` / `MapUtils` 相关 API。
- 集合映射、过滤、分组、归并、去重等操作，在可读性更好、复杂度可控的前提下，应优先使用 `Stream` API；若 `Stream` 明显降低可读性，则回退普通循环。
- 对象判空、默认值提供、非空约束等，必须优先使用 `Objects.isNull(...)`、`Objects.nonNull(...)`、`Objects.requireNonNull(...)`、`Objects.requireNonNullElse(...)` 等 JDK API；不得长期散落风格不一致的手写判空模板。
- `Optional` 只在确实提升语义表达时使用；优先用于方法返回值表达“可能无值”的场景，不得把 `Optional` 当作字段、方法参数或序列化对象的默认容器滥用。
- `Optional` 一旦使用，禁止再让其本身为 `null`。
- Repository 层对外暴露的单实体查询方法必须统一返回 `Optional<T>` 表达“可能无值”，包括 `model.port` 接口以及无对应 port 但作为 repository 对外调用的 concrete repository；禁止返回 `null` 作为正式查询契约。
- Repository 实现内部可接收 mapper / Redis 的空值结果，但必须在 repository 边界转换为 `Optional.empty()`，调用方不得再包一层 `Optional.ofNullable(repository.findXxx(...))`。
- 只要成熟工具类或现成 JDK API 能完整等价替代手写模板代码，必须优先使用，不得新增重复实现。
- MySQL 持久化默认采用 MyBatis-Plus Lambda Wrapper 链式函数写法表达查询、更新、删除条件（`Wrappers.lambdaQuery(...)` / `Wrappers.lambdaUpdate(...)`）。
- `insert(entity)` 作为标准新增写法允许保留。
- 只有 SQL 复杂到链式写法明显不可读时，才允许 XML / 字符串 SQL / 注解 SQL。
- 金额计算必须使用 `BigDecimal`；禁止使用 `double` / `float` 表达金额语义。
- 时间处理统一优先 `java.time` 体系，禁止在新代码中继续扩散 `Date` / `Calendar` 老写法。
- `var` 仅在初始化表达式已足够清晰、变量类型对阅读者显而易见、且使用后能减少噪音时使用；若会弱化类型信息或损害可读性，则禁止使用。

### 3.11 控制流与可读性规范
- 代码应优先追求直接、稳定、可理解的控制流；不要为“写得短”牺牲可读性。
- 条件表达式必须尽量保持单一语义；复杂条件应拆出具名布尔变量或判定方法。
- 对需要明确优先级的布尔表达式，应适度使用括号增强可读性，不依赖阅读者猜测运算优先级。
- `if / else if / else`、`switch`、循环等结构应保持扁平；当嵌套层级过深时必须重构。
- 不允许把大量业务分支直接堆在同一个方法中而不抽取规则语义。
- `switch` 仅用于离散、清晰、可穷举分支；复杂规则应升级为策略、枚举行为或独立协作者。
- 早返回（guard clause）在能明显减少嵌套、提升可读性时应优先使用。
- 不允许使用隐晦的三元表达式、链式 `Optional` 花活或复杂 `Stream` 嵌套替代本应清晰展开的业务逻辑。
- 被调用频率高或关键路径上的日志、异常、控制流分支必须避免无意义对象创建和重复计算。

### 3.12 注释与 Javadoc 规范
- Java 程序中的正式文档注释统一使用 `/** ... */` doc comments；临时说明、实现细节说明使用行注释或块注释。
- 注释必须解释“为什么这样做”或“边界/约束/风险是什么”，不得机械重复代码字面含义。
- 公共类、公共接口、公共方法、对外契约对象、关键枚举与复杂规则点，应补充 Javadoc 或等价正式说明。
- 包级说明应优先写在 `package-info.java` 中，不得散落在若干随机类头注释中替代包级文档。
- 对外暴露的方法注释应明确参数语义、返回语义、异常边界、线程安全假设和副作用。
- 注释必须与代码同步维护；代码语义已变化时，旧注释必须同步更新或删除。
- 禁止使用“TODO 先这样”“临时修复”“后面再看”这类无上下文、无责任人、无明确归属的注释长期留在主干。
- 对明显可通过命名表达清楚的简单逻辑，不得堆砌无价值注释制造噪音。

### 3.13 异常与错误码规范
- `ErrorCode` 命名必须单义、可扩展。
- 错误码粒度要按业务对象区分，不允许 transcript / state / summary 共用一个模糊错误码。
- 统一异常转换只能放在 `app` 层 advice；不要在各层自行做 HTTP 协议转换。
- 业务异常统一继承项目标准异常基类；不允许随处 `throw new RuntimeException(...)` 充当正式错误边界。
- 异常与错误码要能帮助定位对象、动作、失败原因，而不是只表达“失败了”。
- 不得吞异常、空 `catch`、只打印日志不处理、或打印后继续按成功路径执行。
- 记录异常日志时必须保留必要上下文；禁止只打一条“执行失败”而没有对象标识和动作信息。
- 对可预期业务异常与不可预期系统异常必须区分处理，避免全部混为 500 级系统故障语义。

### 3.14 并发与线程安全规范
- 涉及共享状态、缓存、上下文容器、惰性初始化、并发更新时，必须显式说明线程安全假设，不得默认假设单线程环境永远成立。
- 非线程安全对象不得在多线程环境中无约束共享。
- 并发控制必须优先选择语义清晰的方案；禁止在没有边界说明的情况下随意混用 `synchronized`、锁对象、原子类和并发容器。
- 线程安全设计必须覆盖可见性、原子性和有序性，不得只“加了锁就算完成”。
- 阻塞操作、外部 I/O、远程调用不得无边界地放入关键锁范围内。
- 若一个类被设计为非线程安全，必须通过命名、注释或使用边界明确告知调用方。

### 3.15 测试与可测试性规范
- 生产代码设计必须考虑可测试性；复杂逻辑不得严重耦合到静态工具、全局状态或框架细节导致无法单测。
- 测试类命名、测试方法命名必须能直接表达被测对象、场景和期望。
- 单元测试应优先覆盖核心规则、边界条件、异常分支和历史缺陷回归场景。
- 集成测试只验证真实模块协作与关键基础设施交互，不得把本应单测解决的问题全部推给集成测试。
- Mock 只用于隔离边界，不得通过过度 Mock 掩盖真实设计缺陷。
- 测试代码也必须遵守基本命名、分层、可读性与重复逻辑治理规范。
- 发现旧测试与新架构冲突时，优先更新或删除不再成立的旧测试，不为了兼容旧测试保留过时逻辑。

### 3.16 占位 / 预留能力规范
- 预留接口或预留类进入主代码前，必须有文档依据。
- 未启用的占位类，必须：
    - 明确标记预留；或
    - 放入 `future` / `reserved` 包；或
    - 在根文档、阶段文档或模块文档中有明确归属说明。
- 无文档、无调用、无明确归属的占位类不得保留。
- 如果某项能力暂不启用，优先删除，确有需要时再加。

### 3.17 代码评审硬规则（长期有效）
- 新增模块依赖必须先过分层审查。
- 新增依赖版本必须先回写父 POM。
- 新增类必须检查包位置是否语义正确。
- 新增方法必须检查参数数量与构造方式是否合理。
- 新增对象必须检查是否应优先不可变。
- 新增枚举必须检查其是否真的是有限、稳定、可穷举的业务集合。
- 新增公共方法、公共工具、公共抽象前，必须先确认是否真有至少两个以上正式使用点支撑抽象成立。
- 新增预留能力必须检查是否突破已确认范围边界。
- 新增或修改阶段性能力时，必须优先更新当前阶段 `execution-phase/{phase-name}/` 下的阶段文档；只有长期稳定规则才允许晋升到根目录文档。
- 所有评审结论必须能明确归因到分层、职责、边界、可维护性、范围一致性中的至少一项，禁止只给空泛意见。

## 4. 文档治理规则（强约束）

根目录四个核心文档、模块级 `AGENTS.md`、`execution-phase/` 阶段文档均属于项目治理资产。后续任何更新都必须遵守以下规则。

### 4.1 文档职责边界

文档职责固定如下：

| 文档区域 | 职责 |
|---|---|
| 根目录 `AGENTS.md` | 仓库级协作规则、通用开发规范、文档治理规则、长期有效的开发约束 |
| 根目录 `ARCHITECTURE.md` | 长期架构分层、模块边界、依赖方向、主调用关系 |
| 根目录 `CODE_REVIEW.md` | 项目级通用审查标准、拒绝项、检查清单与测试门禁 |
| 根目录 `PROJECT_PLAN.md` | 高层路线图、阶段状态、当前阶段索引 |
| 模块级 `AGENTS.md` | 模块职责、包边界、本模块长期开发约束 |
| `execution-phase/{phase-name}/` | 当前阶段详细设计、开发计划、阶段测试验收、Codex prompt、阶段收口 |

根目录文档负责长期治理。  
模块级文档负责模块边界。  
阶段目录文档负责阶段细节。

### 4.2 `execution-phase/` 阶段文档规则

从 P2-E 开始，阶段性详细设计、开发计划、测试验收、Codex prompt 与阶段收口记录，统一放入：

```text
execution-phase/{phase-name}/
```

每个新阶段建议使用以下结构：

```text
execution-phase/{phase-name}/
├── README.md
├── design.md
├── plan.md
├── test.md
├── prompts.md
└── closure.md
```

各文件职责如下：

| 文件 | 职责 |
|---|---|
| `README.md` | 当前阶段入口说明、阶段目标摘要、阅读顺序、关键文件索引 |
| `design.md` | 当前阶段详细系统设计、领域对象、包边界、强契约、流程设计、不做事项 |
| `plan.md` | 当前阶段开发计划、任务拆分、输入输出、修改范围、执行顺序、完成标准 |
| `test.md` | 当前阶段 code review 清单、回归测试范围、反回退检查项、验收标准 |
| `prompts.md` | 当前阶段给 Codex 的正式 prompt、执行目标、执行结果摘要 |
| `closure.md` | 当前阶段最终收口记录、完成范围、遗留事项、是否需要晋升到根文档 |

### 4.3 文档优先级规则

当多个文档之间存在重叠或冲突时，按照以下优先级判断：

```text
1. 当前阶段 execution-phase/{phase}/design.md 中明确声明的新增阶段强契约
2. 当前阶段 execution-phase/{phase}/plan.md 和 test.md
3. 已冻结的历史阶段强契约文档
4. 根目录 AGENTS.md / ARCHITECTURE.md / CODE_REVIEW.md / PROJECT_PLAN.md
5. 模块级 AGENTS.md
```

补充规则：

- 已有对象优先遵守历史强契约；
- 新增对象优先遵守当前阶段 `design.md`；
- 无法判断属于已有对象还是新增对象时，必须先修正文档，再继续实现；
- 不允许 Codex 或开发者用工程习惯自行推断、改名、扩展字段。

### 4.4 根目录文档更新规则

根目录文档只在以下情况更新：

1. 某个阶段产出的规则已经成为长期通用规则；
2. 某个阶段引入了稳定的新架构层；
3. 某个阶段改变了长期模块边界；
4. 当前阶段状态需要在 `PROJECT_PLAN.md` 中更新；
5. 文档优先级、协作规则、review 总原则发生变化。

阶段详细设计不应写入根目录 `ARCHITECTURE.md`。  
阶段完整开发计划不应写入根目录 `PROJECT_PLAN.md`。  
阶段专项 review 清单不应写入根目录 `CODE_REVIEW.md`。  
阶段 Codex prompt 不应写入根目录 `AGENTS.md`。

这些内容应写入对应的：

```text
execution-phase/{phase-name}/
```

### 4.5 阶段文档冻结规则

每个阶段完成后，必须通过 `closure.md` 进行收口。

阶段完成后：

- `design.md` 不应被随意重写；
- `plan.md` 不应被反复改造成新阶段计划；
- `test.md` 不应被删除关键验收项；
- `prompts.md` 应保留真实执行过的 Codex prompt；
- `closure.md` 应记录最终完成范围、遗留事项和是否晋升根文档。

如果阶段完成后发现设计需要补充，不建议直接重写原始设计内容。  
推荐在阶段文档末尾追加 `Design Amendments` 或 `Post-closure Notes`，用于保留文档演进痕迹。

### 4.6 基线规则

任何文档更新必须以当前文件内容为基线进行增量更新。

不涉及变动的内容不得改写。  
未经确认，不得重写文件整体风格。  
未经确认，不得擅自改变布局、排版、标题层级、写法、风格、章节顺序。

### 4.7 允许的修改范围

允许的修改包括：

- 在原有章节内补充必要内容；
- 新增当前确实需要的新章节；
- 更新日期、默认基线、阶段索引等必要信息；
- 删除已明确确认废弃且必须移除的旧约束；
- 将阶段性细节下沉到 `execution-phase/{phase-name}/`；
- 将模块内长期边界下沉到模块级 `AGENTS.md`。

### 4.8 禁止事项

禁止以下行为：

- 把原文档整体改写成另一种风格；
- 把原本职责清晰的文件改写成职责混乱的综合说明书；
- 把阶段详细设计长期堆叠到根目录文档中；
- 把阶段专项 review 清单长期堆叠到根目录文档中；
- 把阶段完整开发计划长期堆叠到根目录文档中；
- 让 Codex 在没有阅读当前阶段文档的情况下直接实现；
- 为了旧测试保留过时架构逻辑；
- 阶段完成后删除关键设计痕迹；
- 新旧阶段语义混写在同一个历史设计文档中；
- 在没有 `closure.md` 收口记录的情况下宣布阶段正式完成；
- 用工程习惯替代冻结文档中的 class 契约；
- 在未明确文档优先级的情况下处理冲突。

## 5. 全局依赖方向

详细架构与调用关系见 `ARCHITECTURE.md`，此处只保留仓库级原则：

- `app` 是唯一运行入口，只做装配、接入、SSE 适配与异常出口。
- `runtime` 是唯一主链路编排中心，但内部协作者必须按职责拆分。
- `infra` 只做实现与适配，不做流程主导。
- `model` 承载内部运行时模型与跨层共享契约。
- `common` 只承载轻量公共能力。
- 若某个契约需要被 `runtime` 与 `infra` 同时依赖，应下沉到 `model`（少量纯基础契约才可放 `common`）。
- `model.port` 只保留接口；`model.provider`、`model.context`、`model.memory`、`model.prompt` 等按真实对象语义拆分。

## 6. 模块 `AGENTS.md` 规范

当前仓库已固定存在以下模块级治理文档：

- `vi-agent-core-app/AGENTS.md`
- `vi-agent-core-common/AGENTS.md`
- `vi-agent-core-infra/AGENTS.md`
- `vi-agent-core-model/AGENTS.md`
- `vi-agent-core-runtime/AGENTS.md`

模块 `AGENTS.md` 负责：

- 模块职责与边界
- 模块内包结构约定
- 局部开发约束
- 测试要求
- 依赖与 POM 的模块级规则

根目录文档、模块文档与阶段文档的职责边界必须固定：

- 根目录文档：仓库级规则、总体架构、审查门禁、通用开发规范
- 模块文档：模块内包结构、类职责、局部约束、模块测试与依赖规则
- 阶段文档：阶段详细设计、阶段开发计划、阶段专项验收、Codex prompt、阶段收口记录

## 7. 一句话总结

`AGENTS.md` 的职责，是把仓库级通用开发规范、依赖方向、POM 原则、文档治理规则和长期职责边界定死；与阶段执行清单、阶段详细设计、阶段 Codex prompt 相关的内容不得混写到本文件。
